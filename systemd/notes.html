What is systemd?

systemd is Linux's modern service manager that replaces older systems like init or upstart.

Starts services during boot
Stops services gracefully
Restarts services automatically
Tracks logs with journalctl
Handles service dependencies
Manages permissions and users

Key Concept: systemd converts your Python script into a managed Linux service with enterprise features.

Step-by-Step Implementation

1

Manual Testing First

Why Test Manually?

Always test your app manually before creating a service:

  • If it fails manually, systemd won't fix it
  • Verify the application logic works
  • Check for missing dependencies
  • Confirm the correct port is open

Testing Commands

$ cd /home/amit/Desktop/Courses/systemd/opt/myapi
$ python3 app.py
# Keep terminal open, open another terminal
$ curl http://localhost:8000
Systemd Professional Service Running!
# Or open browser to http://localhost:8000
Critical: Never skip manual testing. Systemd will only run what you give it.
2

Create Service File

Service File Location

Systemd service files are stored in:

# User-specific services (rarely used)
~/.config/systemd/user/

# System-wide services (recommended)
/etc/systemd/system/

We'll use /etc/systemd/system/ for system-wide service.

Create the File

$ sudo nano /etc/systemd/system/myapi.service
# Or use vim, vi, or any text editor
# Requires sudo since it's in /etc

Filename convention: servicename.service

3

Write Service Configuration

# /etc/systemd/system/myapi.service
[Unit]
Description=My API Service
After=network.target

[Service]
User=amit
WorkingDirectory=/home/amit/Desktop/Courses/systemd/opt/myapi
ExecStart=/usr/bin/python3 /home/amit/Desktop/Courses/systemd/opt/myapi/app.py
Restart=always

[Install]
WantedBy=multi-user.target

[Unit] Section

  • Description: Human-readable name
  • After: Start after network is ready

[Service] Section

  • User: Run as specific user
  • WorkingDirectory: Directory context
  • ExecStart: Command to run
  • Restart: Auto-restart policy

[Install] Section

  • WantedBy: When to start (boot)

Service File Deep Dive

[Unit] Section Analysis

[Unit]
Description=My API Service
After=network.target

Description

Just a human-readable name shown in systemctl status. Doesn't affect functionality.

After=network.target

This is a dependency directive:

  • Ensures network is ready before starting
  • Critical for web servers that bind to ports
  • Without this, service might fail if network isn't ready
# Other common dependencies:
After=network-online.target
After=syslog.target
After=postgresql.service # If you need database first

[Service] Section Analysis

User=amit

Security Best Practice: Never run services as root unless absolutely necessary.

# Bad - runs as root (insecure)
User=root

# Good - runs as specific user
User=amit

# Better - create dedicated service user
User=myapi-user

If compromised, damage is limited to user permissions.

WorkingDirectory

Sets the working directory before running ExecStart.

Common Error: Wrong path causes status=200/CHDIR error

ExecStart Deep Explanation

ExecStart=/usr/bin/python3 /home/amit/Desktop/Courses/systemd/opt/myapi/app.py

Absolute Paths Required

Systemd requires full absolute paths for:

  • Python interpreter (/usr/bin/python3)
  • Script location (full path to app.py)
# WRONG - relative path
ExecStart=python3 app.py

# CORRECT - absolute path
ExecStart=/usr/bin/python3 /full/path/to/app.py

Restart Policies

always Restart always (default for production)
on-success Only restart on clean exit
on-failure Only restart on failure
no Never restart

With Virtual Environment

# If using virtual environment
ExecStart=/home/amit/Desktop/Courses/systemd/opt/myapi/venv/bin/python3 app.py

[Install] Section & Boot Process

WantedBy=multi-user.target

This creates a symbolic link when you run systemctl enable:

/etc/systemd/system/
├── myapi.service
└── multi-user.target.wants/
    └── myapi.service → ../myapi.service

During boot, systemd activates multi-user.target, which starts all services that "want" it.

Boot Process Flow

  1. System boots, systemd initializes
  2. Reaches multi-user.target stage
  3. Checks for services in .wants/ directory
  4. Starts myapi.service
  5. Runs ExecStart command
  6. Service runs in background

Result: Your Python app starts automatically on every boot without manual intervention.

Essential systemctl Commands

Service Lifecycle

sudo systemctl daemon-reload
# Reload systemd after modifying service files
# REQUIRED after changing .service file
sudo systemctl enable myapi
# Enable service to start on boot
# Creates symlink in .wants/ directory
sudo systemctl start myapi
# Start service immediately
# Doesn't enable on boot
sudo systemctl restart myapi
# Restart service (stop then start)
# Useful after code changes

Monitoring & Status

sudo systemctl status myapi
# Most important command
# Shows: running state, errors, process ID
systemctl is-enabled myapi
# Check if service is enabled for boot
# Returns: enabled, disabled, or static
systemctl is-active myapi
# Check if service is currently running
# Returns: active, inactive, or failed
sudo systemctl list-units --type=service
# List all system services
# | grep myapi to filter

Log Management with journalctl

Viewing Logs

journalctl -u myapi
# View all logs for myapi service
journalctl -u myapi -f
# Follow logs in real-time (like tail -f)
journalctl -u myapi --since "10 minutes ago"
# View logs from last 10 minutes
journalctl -u myapi -n 50
# Show last 50 log entries

Advanced Log Filtering

journalctl -u myapi --no-pager
# Output without pager (for scripts)
journalctl -u myapi -p err
# Show only error messages
journalctl --since today --until "1 hour ago"
# Time-based filtering
sudo journalctl --disk-usage
# Check journal log disk usage

Tip: Journal logs are stored in binary format and persist across reboots by default.

Troubleshooting Common Errors

status=200/CHDIR Error

Symptoms

$ sudo systemctl status myapi
● myapi.service - My API Service
  Loaded: loaded (/etc/systemd/system/myapi.service; enabled)
  Active: failed (Result: exit-code)
  Process: 12345 ExecStart=... (code=exited, status=200/CHDIR)

CHDIR = Change Directory failed.

Root Cause & Fix

Cause: Wrong path in WorkingDirectory

# Wrong path example
WorkingDirectory=/home/amit/Desktop/systemd/opt/myapi

# Correct path example
WorkingDirectory=/home/amit/Desktop/Courses/systemd/opt/myapi
Fix: Check path exists: ls -la /home/amit/Desktop/Courses/systemd/opt/myapi

Permission Denied Errors

Symptoms

$ sudo systemctl status myapi
● myapi.service - My API Service
  Loaded: loaded (/etc/systemd/system/myapi.service; enabled)
  Active: failed (Result: exit-code)
  Process: 12345 ExecStart=...
  Main PID: 12345 (code=exited, status=203/EXEC)
Permission denied

Root Causes

  • Python file not executable: chmod +x app.py
  • Wrong ownership: chown amit:amit app.py
  • User in service file doesn't have permission
  • Virtual environment permissions incorrect
# Check permissions
ls -la /home/amit/Desktop/Courses/systemd/opt/myapi/app.py

# Check python executable
ls -la /usr/bin/python3

Debugging Checklist

1. Test Manually

$ cd /path/to/app
$ python3 app.py
# Does it work manually?

2. Check Service File

$ sudo systemctl cat myapi
# View the actual service file
$ sudo systemctl edit myapi
# Edit service file (creates override)

3. View Full Logs

$ journalctl -u myapi -xe
# Extended logs with details
$ journalctl -u myapi --no-pager | tail -50

Pro Tip: Run as ExecStartPre

# Add to [Service] section for debugging
ExecStartPre=/bin/echo "Starting myapi service at $(date)"
ExecStartPre=/usr/bin/whoami
ExecStartPre=/bin/pwd

These commands run before your main command and appear in logs.

Service Management & Control

Stopping the Service

Stop Immediately

sudo systemctl stop myapi
# Stops service right now
# Keeps enabled for next boot
# Frees port 8000

Verify:

$ sudo systemctl status myapi
Should show: inactive (dead)

Disable Auto-Start

sudo systemctl disable myapi
# Prevents auto-start on boot
# Service remains stopped
# Can still be started manually

Verify:

$ systemctl is-enabled myapi
Should show: disabled

Complete Service Removal

Step-by-Step Removal

# 1. Stop the service
sudo systemctl stop myapi

# 2. Disable auto-start
sudo systemctl disable myapi

# 3. Remove service file
sudo rm /etc/systemd/system/myapi.service

# 4. Reload systemd
sudo systemctl daemon-reload

# 5. Reset failed state (if any)
sudo systemctl reset-failed

Quick Reference

Action Command
Stop now sudo systemctl stop myapi
Disable boot start sudo systemctl disable myapi
Stop + Disable sudo systemctl stop myapi && sudo systemctl disable myapi
Delete permanently remove service file + daemon-reload

Advanced Management Scenarios

Service Overrides

Create override without modifying original service file:

sudo systemctl edit myapi
# Creates: /etc/systemd/system/myapi.service.d/override.conf
# Add overrides like:
[Service]
Environment="DEBUG=true"
RestartSec=5 # Wait 5 seconds before restart

View overrides:

$ sudo systemctl cat myapi
# Shows original + overrides

Environment Variables

[Service]
Environment="DATABASE_URL=postgresql://user:pass@localhost/db"
Environment="DEBUG=false"
EnvironmentFile=/etc/myapi/env.conf

Access in Python:

import os
db_url = os.getenv('DATABASE_URL')

Best Practice: Use EnvironmentFile for sensitive data like API keys.
