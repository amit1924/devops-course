üéØ Pods per Node: Critical Concept

Question:

Example Calculation

Each Pod requests:

CPU: 200m (0.2 cores)

Memory: 256Mi

Node capacity:

2 CPU cores = 2000m

2Gi memory = 2048Mi

Memory-wise:

2048 / 256 = 8 Pods

CPU-wise:

2000 / 200 = 10 Pods

üß© Multi-Container Pod Patterns

Sidecar Pattern

Regular container running alongside main app for entire Pod lifecycle

Use Cases:

  • TLS encryption (Envoy)
  • Traffic metrics
  • Request tracing
  • Log shipping
spec: containers: - name: express-app # Main container image: express:v1 - name: envoy-sidecar # Sidecar image: envoy:latest

Init Container Pattern

Special containers that run first, once, must complete before main app starts

Use Cases:

  • Wait for database readiness
  • Clone Git repository
  • Config preparation
  • Secrets validation
spec: initContainers: - name: init-db-check image: busybox command: ['sh', '-c', 'until nslookup db; do sleep 2; done'] containers: - name: app image: my-app:latest

Sync + Serve Pattern

One container syncs data, another serves it - both share volume

Example:

  • Container 1: Git sync
  • Pulls to shared volume
  • Container 2: Express server
  • Serves from shared volume
spec: containers: - name: git-sync image: git-sync:latest volumeMounts: - name: shared-data mountPath: /data - name: web-server image: nginx:latest volumeMounts: - name: shared-data mountPath: /usr/share/nginx/html

üíæ Resource Requests & Limits (CRITICAL)

üß† Resource Request

Tells scheduler: "I need at least this much CPU & memory" 

resources:
  requests:
    cpu: "200m"     # 0.2 CPU cores
    memory: "256Mi"  # 256 Megabytes

Scheduler only places Pod on nodes that can satisfy this

üö® Resource Limit

Maximum allowed usage before action is taken

resources:
  limits:
    cpu: "500m"     # Max 0.5 CPU cores
    memory: "512Mi"  # Max 512 MB

If container exceeds:

  • CPU ‚Üí throttled
  • Memory ‚Üí OOMKilled

Why This Is Important

If you don't specify requests:

  • Scheduler guesses nothing
  • Might pack too many Pods on one node
  • Can cause crashes and instability

Production rule:

ALWAYS define requests and limits

Engineering Insight: Without limits, a memory leak can consume entire node memory

üîÅ Restart Policies Explained

üü¢ Always

(Default for most apps)

If container crashes ‚Üí restart
For long-running apps
Express, web servers, databases

üü° OnFailure

Restart only on failure

If job completes ‚Üí don't restart
If job crashes ‚Üí restart
Batch jobs, one-time tasks

üî¥ Never

Fire-and-forget

No restart under any condition
Debug containers
Disposable workloads

Important Clarification

Container Restart:

kubelet restarts container inside same Pod

restartPolicy controls this

Pod Replacement:

If node crashes, controller creates NEW Pod on another node

This is NOT restart, it's replacement
